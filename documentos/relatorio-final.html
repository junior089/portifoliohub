1. Planejamento da Implantação & IA Gemini

O planejamento do processo de implantação foi estruturado para garantir que a transição entre o desenvolvimento local e a publicação em ambiente de produção fosse segura, documentada e livre de falhas de ambiente. Para guiar esta trilha, foi utilizado o **Google Gemini** como consultor de arquitetura e implantação.

Metodologia de Engenharia de Prompt com Gemini

A interação com a Inteligência Artificial foi estruturada usando prompts contextuais e incrementais. O objetivo principal foi criar um cronograma de atividades focado nas melhores práticas de integração contínua (CI) e segurança web. O prompt inicial de planejamento seguiu o padrão:

[PROMPT]
Atue como especialista em DevOps e Segurança de Aplicações. Preciso de um plano
de implantação detalhado para o projeto PortfolioHUB (HTML/CSS/JS estático) que
será publicado no GitHub Pages. Defina a trilha cobrindo: controle de versão,
políticas IAM no GitHub, segurança de cabeçalhos (CSP) e checklist de testes.

A resposta gerou um fluxo sequencial que Carlos Alberto Junior seguiu rigorosamente, dividindo a entrega final em três blocos de execução: infraestrutura (integração GitHub API), governança (configurações de segurança no repositório) e polimento final (testes de responsividade e renderização).

Benefícios da Orientação por IA

  • Redução de Erros: A estruturação antecipada impediu a inclusão acidental de chaves de API públicas no histórico do Git.
  • Foco em Segurança por Design: A recomendação de configurar regras de Branch Protection antes do início do código da Entrega Final evitou pushes inseguros na branch principal.

2. Configuração Inicial e Integração com GitHub (API)

A base da infraestrutura foi estabelecida em um repositório Git público dedicado no GitHub. A evolução em relação à entrega intermediária consistiu em integrar a própria API REST do GitHub no frontend da aplicação, oferecendo uma experiência dinâmica e profissional.

Integração com a API REST do GitHub

No arquivo main.js, implementamos uma função assíncrona chamada loadGitHubStats(). Essa função realiza requisições HTTP do tipo GET para os endpoints da API pública do GitHub, carregando métricas atualizadas sobre o projeto. Isso demonstra a integração do PortfolioHUB com ecossistemas externos de desenvolvimento.

// Estrutura do consumo da API do GitHub no main.js
const repoPath = "junior089/portfoliohub-entrega-intermediaria";
const apiURL = `https://api.github.com/repos/${repoPath}`;

try {
  const repoRes = await fetch(apiURL);
  const repoData = repoRes.ok ? await repoRes.json() : {};
  // Atualiza os contadores na tela (Stars, Forks, Issues, etc.)
  document.getElementById("github-stars").textContent = repoData.stargazers_count;
} catch (error) {
  console.warn("Utilizando fallbacks devido a limites da API pública");
}

Estratégia de Fallback

Para garantir que a página continue funcional mesmo sob limites de requisições da API pública (Rate Limiting), foi implementada uma estrutura de tratamento de exceções (try/catch). Em caso de falha de conexão ou HTTP 403, dados pré-configurados e realistas são carregados em tela de forma transparente para o usuário.

3. Gestão de Usuários, Controle de Acesso (IAM) e Segurança

A governança e a segurança da informação foram implementadas em conformidade com as diretrizes da atividade, utilizando o princípio do menor privilégio (PoLP) e blindagem contra ataques comuns (OWASP Top 10).

Gestão de Usuários e Permissões IAM

A atribuição de controle de acesso foi planejada para simular um ambiente corporativo dentro do GitHub:

  • Owner (Carlos Alberto Junior): Acesso administrativo total, gerencia configurações, chaves de criptografia e faturamento.
  • Collaborator (Avaliadores/Colegas): Permissão de escrita restrita. Alterações só podem ser introduzidas por meio de Branches de Feature e Pull Requests, com aprovação obrigatória.
  • Reader / Auditor (Público Acadêmico): Acesso de leitura ao código-fonte, garantindo a transparência do trabalho.

Políticas de Segurança do Repositório

Seguindo as orientações do assistente Gemini, as seguintes configurações de proteção ativa foram ativadas no repositório:

  1. Branch Protection: A branch main foi configurada como protegida. Ela não aceita commits diretos (exigindo Pull Request), necessita de aprovação prévia de revisores e bloqueia Force Pushes (git push --force).
  2. Secret Scanning & Dependabot: Ferramentas automatizadas do GitHub que monitoram a inclusão acidental de chaves privadas e avisam sobre dependências vulneráveis.
  3. Content Security Policy (CSP): Configuração de uma meta tag de política de segurança de conteúdo na página principal para evitar ataques de injeção de scripts (XSS):
<meta http-equiv="Content-Security-Policy" 
      content="default-src 'self'; 
               script-src 'self' 'unsafe-inline' https://api.github.com; 
               style-src 'self' 'unsafe-inline'; 
               connect-src 'self' https://api.github.com;">

4. Compartilhamento, Fluxo de Trabalho (GitFlow) e Colaboração

Para gerenciar o desenvolvimento da Entrega Final sem comprometer a estabilidade do site publicado na entrega intermediária, foi adotado um modelo simplificado de GitFlow e padronização semântica de histórico.

Modelo de Branches Utilizado

O fluxo de compartilhamento de código foi baseado na separação de responsabilidades:

  • Branch `main`: Ramificação estável de produção. Reflete exatamente o site público hospedado no GitHub Pages. Só recebe código testado e aprovado através de Pull Requests da branch de desenvolvimento.
  • Branch `develop` (ou features locais): Utilizada para o desenvolvimento das novas funcionalidades da Entrega Final (Widget Gemini, segurança, API).

Padrão de Commits Semânticos (Conventional Commits)

Cada commit da Entrega Final foi estruturado com prefixos claros para facilitar a auditoria do progresso:

- `feat: adiciona widget interativo do assistente Gemini e lógica do chat`
- `feat: consome API do GitHub para exibir estatísticas em tempo real`
- `security: adiciona meta tag de Content Security Policy (CSP) na home`
- `docs: atualiza relatorio final de implantacao com as seis seções da entrega`
- `chore: atualiza slides de apresentacao e cria roteiro final de video`

Práticas de Colaboração e Code Review

A colaboração foi documentada com a criação de um modelo de Pull Request que exige do desenvolvedor preencher uma descrição das alterações feitas, os testes manuais realizados e as regras de segurança validadas antes que o código seja mesclado na branch principal.
