Slide 05 • IAM

4. Gestão de Usuários e Controle de Acesso (IAM)

Modelagem de papéis e controle de privilégios no ecossistema do GitHub para colaboração segura.

  • Owner: Acesso administrativo completo às chaves de criptografia e deploy.
  • Collaborator: Permissões de escrita restritas via Pull Requests e aprovação de revisores.
  • Reader / Auditor: Permissão de leitura pública para conformidade e transparência acadêmica.
Roteiro de Fala (Tempo sugerido: 30s) "A gestão de usuários e controle de acesso foi espelhada em um ambiente profissional corporativo. Configuramos a gestão do GitHub dividindo as permissões em: Owner para administração geral, Collaborators para desenvolvedores com permissão de escrita via Pull Request, e Readers para o público examinador. Isso garante que o código principal não seja modificado sem supervisão."
Slide 06 • Segurança

5. Políticas de Segurança e OWASP recomendadas por IA

Implementação de regras de proteção e auditorias ativas contra vulnerabilidades.

  • Branch Protection: Exigência de Pull Requests, aprovações, commits assinados e bloqueio de pushes diretos na main.
  • Content Security Policy (CSP): Meta tag de segurança restringindo fontes de scripts para combater ataques de XSS.
  • Security Scanning: Monitoramento de dependências via Dependabot e Secret Scanning de chaves.
Roteiro de Fala (Tempo sugerido: 40s) "No quesito segurança, aplicamos três políticas essenciais sugeridas pelo Gemini. Primeiro, as Branch Protection Rules na main, bloqueando commits diretos ou force push. Segundo, adicionamos uma meta tag de Content Security Policy, que evita injeção de scripts maliciosos ao restringir fontes externas. E terceiro, ativamos o Dependabot e o Secret Scanning do GitHub para auditar vulnerabilidades e evitar vazamentos de chaves."
Slide 07 • Versionamento

6. Controle de Versão e Práticas de Colaboração

Uso estruturado do GitFlow e tags de versão para controle de entregas.

  • Fluxo GitFlow: Separação estrita de branches: main (produção) e branches locais de features para a entrega final.
  • Commits Semânticos: Histórico limpo e padronizado usando Convenções (feat, security, docs, chore).
  • Tags de Versão: Uso da tag v1.0 para a entrega intermediária e v2.0-final para a implantação final.
Roteiro de Fala (Tempo sugerido: 30s) "Em termos de versionamento e controle de versão, utilizamos o Git de forma disciplinada. O fluxo separa a branch principal main da área de desenvolvimento local. Adotamos o padrão de commits semânticos para marcar precisamente cada avanço técnico, e criamos a tag v2.0-final para congelar a versão definitiva de produção da Entrega Final."
